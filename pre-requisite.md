# Prerequisites

What you need before deploying or using `azure-notification-hub-unofficial`.

---

## 1. Azure Resources

### 1.1 Notification Hub Namespace

An Azure Notification Hubs namespace is the top-level container.

| Resource | Description |
|---|---|
| **Namespace** | Logical container (e.g. `myns.servicebus.windows.net`) |
| **Notification Hub** | Named hub inside the namespace (e.g. `myhub`) |
| **Access Policy** | SAS policy with at minimum **Send** and **Manage** claims |

**Create via Azure Portal:**

1. Go to **Azure Portal → Create a resource → Notification Hub Namespace**
2. Or create via Azure CLI:

```bash
# Create namespace
az notification-hub namespace create \
  --resource-group <rg> \
  --name <namespace-name> \
  --location <region>

# Create hub inside namespace
az notification-hub create \
  --resource-group <rg> \
  --namespace-name <namespace-name> \
  --name <hub-name>

# Get connection string
az notification-hub authorization-rule list \
  --resource-group <rg> \
  --namespace-name <namespace-name> \
  --notification-hub-name <hub-name>
```

**Connection string format:**

```
Endpoint=sb://<namespace>.servicebus.windows.net/;SharedAccessKeyName=<policy>;SharedAccessKey=<key>;EntityPath=<hub-name>
```

### 1.2 Platform Notification Service Credentials

Each platform you target must be configured in the Azure Notification Hub.

| Platform | What to configure in Azure Portal | What you need |
|---|---|---|
| **Apple (APNs)** | Hub Settings → Apple (APNS) | APNs certificate (.p12) or token (Key ID, Team ID, Bundle ID) |
| **FCM Legacy (GCM)** | Hub Settings → Google (GCM) | API Key from Firebase Console → Cloud Messaging |
| **FCM V1** | Hub Settings → Google (FCM V1) | Service Account JSON (Firebase Console → Project Settings → Service Accounts) |
| **Windows (WNS)** | Hub Settings → Windows (WNS) | Package SID, Security Key (Partner Center or Windows Dev Center) |
| **ADM (Amazon)** | Hub Settings → Amazon (ADM) | Client ID, Client Secret (Amazon Developer Portal) |
| **Baidu** | Hub Settings → Baidu | API Key, Secret Key (Baidu Push Console) |
| **Xiaomi** | Hub Settings → Xiaomi | App ID, App Key (Xiaomi Developer Portal) |
| **Browser (Web Push)** | Configured at runtime via the SDK (no portal config needed) | VAPID keys (for push channel subscription) |

> **Note:** Browser (Web Push) does not require Azure-side configuration. VAPID keys are generated by your application server, and the browser's `PushSubscription` is registered via the SDK's `create_or_update_installation()`.

### 1.3 SKU Considerations

| SKU | Scheduled Notifications | Device Count | Features |
|---|---|---|---|
| **Free** | ❌ No | Up to 500 devices | Basic send/receive |
| **Basic** | ❌ No | Unlimited | Tags, templates, per-device auth |
| **Standard** | ✅ Yes | Unlimited | Everything + scheduled sends + analytics |

> `schedule_notification()` and `cancel_scheduled_notification()` require **Standard SKU** or above.

## 3. Network / Firewall Whitelisting

The SDK communicates with Azure Notification Hubs REST API and — indirectly — with each platform's push notification service.

### 3.1 Azure Notification Hubs Endpoint

All API calls go to:

```
https://<namespace>.servicebus.windows.net/
```

**Whitelist** (allow outbound HTTPS / port 443):

| Host | Port | Purpose |
|---|---|---|
| `*.servicebus.windows.net` | 443 (HTTPS) | Notification Hubs REST API |

> The SDK translates `sb://` to `https://` automatically.

### 3.2 Platform Push Service Endpoints (Outbound from Azure)

Azure Notification Hubs delivers notifications to platform services. If your environment uses **proxy, firewall, or egress restrictions**, note that these endpoints are called **from Azure's infrastructure** (not from your code). However, your devices need to reach these services to receive notifications.

| Platform | Push Service Endpoint | Port |
|---|---|---|
| **Apple (APNs)** | `api.push.apple.com` (production) / `api.development.push.apple.com` (sandbox) | 443 (HTTPS) or 2197 |
| **FCM Legacy** | `fcm.googleapis.com` / `fcm-xmpp.googleapis.com` | 443 (HTTPS), 5236 (XMPP) |
| **FCM V1** | `fcm.googleapis.com` | 443 (HTTPS) |
| **Windows (WNS)** | `*.notify.windows.com` | 443 (HTTPS) |
| **ADM (Amazon)** | `dm.amazon.com` | 443 (HTTPS) |
| **Baidu** | `api.push.baidu.com` | 443 (HTTPS) |
| **Xiaomi** | `api.xmpush.xiaomi.com` | 443 (HTTPS) |

> **Important for clients using your SDK:** If the machine running the SDK is behind a corporate firewall, it only needs outbound HTTPS to `*.servicebus.windows.net`. The platform push endpoints are called by Azure, not by the SDK directly.

### 3.3 Browser Web Push (VAPID)

Browser Web Push goes through the browser's native push service. The browser connects to:

| Browser | Push Service |
|---|---|
| Chrome / Edge | `fcm.googleapis.com` (via Firebase Cloud Messaging) |
| Firefox | `updates.push.services.mozilla.com` |
| Safari | `push.apple.com` |

These are contacted by the **end user's browser**, not by your server.

## 4. Permissions / RBAC

### 4.1 Azure Access Policies

The connection string determines what operations the SDK can perform.

| Access Policy | Claims | Can do |
|---|---|---|
| `DefaultFullSharedAccessSignature` | **Manage, Send, Listen** | All operations |
| `DefaultListenSharedAccessSignature` | Listen only | Read-only (get registrations, feedback) |
| Custom policy with **Send** | Send only | Send notifications, create/update installations |
| Custom policy with **Manage** | Manage + Send + Listen | Full access |

**Minimum policy for typical usage:** `DefaultFullSharedAccessSignature` or a custom policy with **Manage** (+ **Send** + **Listen** claims).

### 4.2 Azure RBAC (Optional)

For managing the Notification Hub itself (create/delete/modify), you need Azure RBAC roles:

| Role | Scope | Purpose |
|---|---|---|
| **Contributor** | Resource group or namespace | Create / delete namespaces and hubs |
| **Owner** | Resource group or namespace | Full management including RBAC |
| **Reader** | Resource group or namespace | View existing hubs |

> These are only needed if your deployment script creates Azure resources. The SDK itself only uses the connection string.

### 4.3 Deployment Permissions

If deploying via CI/CD (GitHub Actions, Azure DevOps, etc.):

- **Service Principal** with `Contributor` on the target resource group, or
- **Managed Identity** assigned to the compute environment (e.g. App Service, AKS).

## 5. Hub Configuration Checklist

Before writing code, ensure these steps are done in the Azure Portal:

- [ ] Notification Hub Namespace created
- [ ] Notification Hub created inside the namespace
- [ ] Access policy configured (connection string obtained)
- [ ] Platform credentials added (for each platform you target)
  - [ ] Apple APNs certificate or token uploaded
  - [ ] FCM / FCM V1 credentials configured
  - [ ] WNS credentials configured
  - [ ] ADM / Baidu / Xiaomi credentials configured (if needed)
- [ ] SKU verified: **Free** (≤500 devices), **Basic** / **Standard** (scheduled sends)

## 6. Development Environment Quick Check

Run this to verify your setup end-to-end:

```python
from azure_notification_hub import NotificationHubsClient

client = NotificationHubsClient(
    connection_string="Endpoint=sb://<ns>.servicebus.windows.net/;SharedAccessKeyName=DefaultFullSharedAccessSignature;SharedAccessKey=<key>;EntityPath=<hub>",
    hub_name="<hub>",
)

# Test: list registrations (returns empty list if no devices registered)
regs = client.list_registrations(top=5)
print(f"Registrations: {regs}")
```

Expected output (empty hub):

```
Registrations: []
```

If you see an `AzureNHError` or timeout, check:

- Connection string is correct (copy from Azure Portal → Access Policies)
- Outbound HTTPS to `*.servicebus.windows.net` is allowed
- Hub name matches the `EntityPath` in the connection string

## 8. Supported Azure Regions

Azure Notification Hubs is available in all Azure regions. For region-specific availability:

```bash
az provider show --namespace Microsoft.NotificationHubs --query "resourceTypes[?resourceType=='namespaces'].locations"
```

**Latency considerations:** Choose a region close to your users and push services.
